# 阶段一：宏观解构 —— LangGraph 智能旅行助手

> 本文档由「金牌技术导师」视角撰写，目标是带你**鸟瞰整个项目**，先建立全局心智地图，再深入细节。
> 配套项目：`LangGraph-trip-planner`（基于 LangGraph 多智能体框架 + 高德地图 MCP 的智能旅行规划助手）。

---

## 0. 一句话概括

> 用户在前端填一张「去哪、玩几天、什么偏好」的表单 → 后端用 **LangGraph 工作流**并行派出 3 个「研究型 Agent」（找景点 / 查天气 / 找酒店）→ 汇总后交给第 4 个「规划型 Agent」生成结构化 JSON 行程 → 增强、校验后返回前端渲染成可视化旅行计划。

这是一个非常典型、也非常适合入门的 **"Orchestrator + Workers"（编排者 + 工人）** 多智能体架构，比单纯的 ReAct 单 Agent 更进一步，又不至于复杂到难以理解。

## 1. 鸟瞰架构：设计模式 & 文件夹角色

### 1.1 整体设计模式

项目在两个层面体现了清晰的设计模式：

| 层面 | 采用的模式 | 说明 |
| --- | --- | --- |
| **系统架构** | 前后端分离 + 分层架构（Layered Architecture） | 前端 Vue3 SPA，后端 FastAPI，通过 REST API 通信 |
| **后端代码** | 经典分层（Routes → Workflow → Agents → Tools/Services） | 每一层只依赖下一层，职责单一 |
| **Agent 编排** | **多智能体「编排-工人」模式 + 状态图（StateGraph）** | 一个图编排多个专职 Agent，状态在节点间流转 |
| **并发模型** | **Fan-out / Fan-in（扇出/扇入）** | 3 个研究 Agent 并行执行，再汇聚到规划节点 |

> 🎓 **导师提示**：很多人学 Agent 时只见过「一个 Agent + 一堆工具」的 ReAct 循环。这个项目的精髓在于——它把"一个全能 Agent"拆成了**多个各管一摊的小 Agent**，再用一张**图（Graph）**把它们的协作关系固化下来。这正是工业级 Agent 从"玩具"走向"可控"的关键一步。

## 2. 核心链路：一次请求的完整生命周期

下面这条 **Pipeline** 是整个项目的灵魂。当用户在前端点击「生成旅行计划」后，数据是这样流动的：

### 2.1 时序总览

```
[用户填表] 
   │  HTTP POST /api/trip/plan  (body = TripRequest)
   ▼
┌─────────────────────────────────────────────────────────┐
│ ① 表现层  api/routes/trip.py :: plan_trip()               │
│    - 接收 TripRequest                                      │
│    - 取得工作流单例 get_trip_planner_workflow()            │
│    - 调用 workflow.plan_trip(request)                     │
└─────────────────────────────────────────────────────────┘
   ▼
┌─────────────────────────────────────────────────────────┐
│ ② 编排层  workflows/trip_planner_graph.py                 │
│    - create_initial_state(request) 构造初始 State          │
│    - self.graph.invoke(initial_state)  ← 进入 LangGraph    │
└─────────────────────────────────────────────────────────┘
   ▼
   ▼  ===== LangGraph StateGraph 内部 =====
   │
   START
   ├──────────────┬──────────────┬──────────────┐   ← Fan-out（并行扇出）
   ▼              ▼              ▼
search_attractions  check_weather  find_hotels
 (景点Agent)        (天气Agent)     (酒店Agent)
   │ 调MCP工具       │ 调MCP工具      │ 调MCP工具
   └──────────────┴──────────────┴──────────────┘
                   ▼   ← Fan-in（扇入汇聚）
          join_research_results  (汇合节点)
                   ▼
          _check_error  (条件边判断)
            ├── "error"  ──► handle_error ──► END  (生成兜底计划)
            └── "continue" ──► plan_itinerary
                                  │ (规划Agent，无外部工具)
                                  ▼ 产出结构化 JSON → 解析为 TripPlan
                               END
   │  ===== 退出 LangGraph =====
   ▼
┌─────────────────────────────────────────────────────────┐
│ ③ 后处理  trip_enhancement_service.enhance_trip_plan()    │
│    - 补全图片、预算、坐标等增强信息                          │
└─────────────────────────────────────────────────────────┘
   ▼
┌─────────────────────────────────────────────────────────┐
│ ④ 返回 TripPlanResponse(success, message, data=TripPlan)  │
└─────────────────────────────────────────────────────────┘
   ▼
[前端 Result.vue 渲染：每日行程 / 地图标记 / 天气 / 餐饮 / 预算]
```

### 2.2 每个节点在做什么（节点 = 一个函数）

在 LangGraph 中，**图的每个节点就是一个普通的 Python 方法**，输入是当前 `State`，输出是要合并进 `State` 的字段字典。

| 节点 | 对应方法 | 职责 | 调用的 Agent |
| --- | --- | --- | --- |
| `search_attractions` | `_search_attractions` | 搜索景点 POI | 景点 Agent → `maps_text_search` |
| `check_weather` | `_check_weather` | 查询天气 | 天气 Agent → `maps_weather` |
| `find_hotels` | `_find_hotels` | 搜索酒店 | 酒店 Agent → `maps_text_search` |
| `join_research_results` | `_join_research_results` | 汇合 3 个并行分支 | 无（仅标记状态） |
| `plan_itinerary` | `_plan_itinerary` | 综合信息生成完整行程 JSON | 规划 Agent（无工具） |
| `handle_error` | `_handle_error` | 任一步出错时生成兜底计划 | 无（本地构造） |

### 2.3 单个研究节点的内部三步走

以 `_search_attractions` 为例，每个研究节点都遵循**「构造查询 → 调 Agent → 解析输出」**三步：

```python
# 1. 构造自然语言查询
query = self._build_attraction_query(state["request"])   # "请搜索北京的历史文化相关景点"

# 2. 调用 Agent（Agent 内部会自主决定调用哪个 MCP 工具）
result = self.attraction_agent.invoke(
    self._prepare_agent_input(query, [])                 # 转成 {"messages": [...]}
)

# 3. 从 Agent 返回的 messages 中抽取文本 → 解析 JSON → 转成 Attraction 对象列表
output = self._extract_agent_output(result)
attractions = self._parse_attractions(output)
return {"attractions": attractions, "messages": [...]}   # 合并回 State
```

> 🎓 **导师提示**：注意这里的"分工"——
> - **Agent 负责"自由发挥"**：理解自然语言、自主决定调哪个工具、调几次（这是 LLM 的 ReAct 部分）。
> - **节点函数负责"收口"**：把 Agent 自由格式的输出，**强行解析成结构化的 Pydantic 对象**（`_parse_attractions` / `_extract_json`）。
>
> 这种「让 LLM 自由探索，但用代码把结果钉成确定结构」的思路，是工业级 Agent 应对 LLM 不确定性的核心手段。

## 3. 技术栈盘点

### 3.1 后端核心栈

| 类别 | 技术 | 在项目中的作用 | 关键文件 |
| --- | --- | --- | --- |
| **Agent 编排框架** | **LangGraph**（`StateGraph`） | 定义多 Agent 协作的图、状态、并行与条件分支 | `trip_planner_graph.py` |
| **Agent/LLM 框架** | **LangChain**（`create_agent`） | 把 LLM + 工具 + 提示词组装成可执行 Agent | `langgraph_agents.py` |
| **LLM 接入** | `langchain_openai.ChatOpenAI` | 统一的 OpenAI 兼容接口（可接 OpenAI/DeepSeek 等） | `llm_service.py` |
| **工具协议** | **MCP**（Model Context Protocol） | 通过 `langchain_mcp_adapters` 加载高德地图工具 | `amap_mcp_tools.py` |
| **外部能力** | **高德地图 amap-mcp-server** | 景点搜索、天气、路线规划 | （通过 `uvx amap-mcp-server` 子进程启动） |
| **Web 框架** | **FastAPI** | 提供 REST API、自动文档、CORS | `api/main.py` |
| **数据校验** | **Pydantic / pydantic-settings** | 请求/响应模型、配置管理 | `schemas.py` / `config.py` |
| **可观测性** | **LangSmith**（可选） | 通过 `langchain_tracing` 开关追踪 Agent 执行 | `config.py` |

### 3.2 前端核心栈

| 类别 | 技术 |
| --- | --- |
| 框架 | Vue 3 + TypeScript |
| 构建工具 | Vite |
| UI 库 | Ant Design Vue |
| HTTP | Axios（`services/api.ts`） |
| 地图渲染 | 高德地图 JavaScript API |

### 3.3 关键技术点辨析

1. **LangGraph vs LangChain 不是二选一，而是分工**：
   - LangChain 的 `create_agent` 负责造**单个** Agent（LLM + 工具 + 提示词的 ReAct 执行器）；
   - LangGraph 的 `StateGraph` 负责把**多个** Agent 编排成一张协作图。
   - 类比：LangChain 造"工人"，LangGraph 当"工头"排班。

2. **MCP（Model Context Protocol）是一大亮点**：
   - 项目没有手写高德 API 调用，而是通过 `uvx amap-mcp-server` 启动一个标准 MCP 子进程，用 `load_mcp_tools` 把它的能力**自动转成 LangChain 工具**。
   - 好处：工具与 Agent 解耦，换一个 MCP 服务器就能换一套能力，符合工业级"可插拔"理念。

3. **三层兜底（健壮性设计）**，体现工程成熟度：
   - 工具加载：真实 MCP → 精简工具 → **模拟工具**（`create_mock_tools`），保证没 Key 也能跑通流程；
   - 流程执行：任一研究节点报错 → `handle_error` 节点 → 生成 **fallback 兜底计划**；
   - JSON 解析：`_parse_trip_plan` 失败 → 同样回退到兜底计划。

## 4. 本阶段小结 & 心智地图

把整个项目浓缩成三句话记住：

1. **架构**：前后端分离 + 后端经典分层；Agent 层用 LangGraph 编排「3 个并行研究 Agent + 1 个规划 Agent」的「编排-工人」模式。
2. **链路**：`HTTP 请求 → 构造 State → 图并行跑研究节点 → 汇合 → 规划节点出 JSON → 解析+增强 → 返回前端`。
3. **技术栈**：LangGraph（编排）+ LangChain（造 Agent）+ MCP（高德工具）+ FastAPI（接口）+ Pydantic（数据），前端 Vue3。
