## 1. 最小可行性 Demo（MVP）：到底要配几个 Key？

### 1.1 ★ 先搞懂一个大坑：环境变量"别名映射" ★

这是本项目**最容易劝退新手**的一点。你打开 `backend/.env.example`，看到的是 `LLM_*` 开头的变量：

```bash
LLM_MODEL_ID="deepseek-chat"
LLM_API_KEY=your key
LLM_BASE_URL=https://api.deepseek.com/v1
```

但你去翻 `llm_service.py`，它读的却是 `OPENAI_API_KEY` / `OPENAI_BASE_URL` / `OPENAI_MODEL`（`llm_service.py:18`）。**名字对不上，为什么能跑？**

答案在 📄 `config.py:23` 的 `_set_openai_aliases_from_llm_env()`：它在启动时做了一次"翻译"——

| `.env` 里写的（旧名） | 自动映射成（代码用的） |
| --- | --- |
| `LLM_API_KEY` 或 `DEEPSEEK_API_KEY` | → `OPENAI_API_KEY` |
| `LLM_BASE_URL` | → `OPENAI_BASE_URL` |
| `LLM_MODEL_ID` | → `OPENAI_MODEL` |

> 🎓 **导师提示**：所以你既可以用 `LLM_API_KEY`（推荐，跟 example 一致），也可以直接写 `OPENAI_API_KEY`，两者等价。**但只有在 `OPENAI_API_KEY` 没值时，别名才会生效**（`if os.getenv(target): continue`）——别两个都写还写矛盾的值。
> 命名为 "OPENAI" 不代表只能用 OpenAI：因为底层是 `ChatOpenAI` + 自定义 `base_url`，**任何 OpenAI 兼容接口（DeepSeek、Kimi、本地 vLLM…）都能接**。

### 1.2 最少必须配的两项（否则启动直接报错）

启动时 `validate_config()`（`config.py:106`）会硬性检查，缺了就 `raise` 拒绝启动：

| 变量 | 是否必须 | 说明 |
| --- | --- | --- |
| `LLM_API_KEY`（→`OPENAI_API_KEY`） | ✅ **必须** | 没有它 `validate_config` 直接报错；这是 Agent 思考的大脑 |
| `AMAP_API_KEY` | ✅ **必须**（为通过校验） | `validate_config` 也会检查它 |
| `LLM_BASE_URL` / `LLM_MODEL_ID` | 🔶 推荐 | 用 DeepSeek 等非 OpenAI 时必填，否则默认连 `api.openai.com` |
| `UNSPLASH_ACCESS_KEY` | ⚪ 可选 | 只影响景点配图，不配也能跑 |
| 前端 `VITE_AMAP_WEB_KEY` 等 | ⚪ 仅前端 | 只影响前端地图渲染，跑后端逻辑用不到 |

> 🎓 **导师提示（重要的"作弊技巧"）**：还记得阶段二讲的**三层兜底**吗？工具加载失败会自动降级到 `create_mock_tools()`（返回假数据）。
> 所以**即使你的 `AMAP_API_KEY` 是无效的占位串**，只要它"非空"能骗过 `validate_config`，工作流照样能用模拟数据跑通全流程！**这意味着你只需要一个真实可用的 `LLM_API_KEY`，就能完整调试整条链路**——这是入门时最省事的起点。

### 1.3 跑通的最小步骤

```bash
# ① 后端
cd backend
python -m venv venv && source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env
# 编辑 .env：至少填好 LLM_API_KEY / LLM_BASE_URL / LLM_MODEL_ID，
# AMAP_API_KEY 随便填个非空值即可先跑通（会走 mock 工具）

python run.py            # 或 uvicorn app.api.main:app --reload --port 8000
```

启动成功的标志：控制台打印配置表 + `[OK] 配置验证通过` + `API文档: http://localhost:8000/docs`。

```bash
# ② 用 Swagger 直接验证（不用碰前端）
# 浏览器打开 http://localhost:8000/docs → POST /api/trip/plan → Try it out → 用示例 body 执行
```

> ⚠️ **一个隐藏依赖**：真实 MCP 工具是用 `uvx amap-mcp-server` 启动子进程的（`amap_mcp_tools.py:117`）。`uvx` 来自 `uv`（已在 requirements 里）。**如果 `uvx` 不在 PATH 或拉取 server 失败，会自动降级到 mock 工具**——这通常不影响你跑通，但能解释"为什么我配了真 Key 拿到的还是假数据"。

## 2. 关键断点设置：在哪儿打断点能看清"思考过程"

调试 Agent 的核心诉求是**看清三件事**：①请求怎么进来 ②Agent 调了什么工具/返回了什么 ③自由文本怎么变成结构化数据。按这条链路，推荐 6 个断点（按执行顺序）。

### 断点地图（从外到内）

```
① api/routes/trip.py:50         plan_trip()            ← 请求入口，看 TripRequest
② trip_planner_graph.py:614     self.graph.invoke()    ← 进入 LangGraph 前的初始 State
③ trip_planner_graph.py:173     _search_attractions    ← 单个研究节点：query 构造 & agent 调用
④ trip_planner_graph.py:79      _extract_agent_output  ← ★ Agent 原始返回的 messages（含 tool_call！）
⑤ trip_planner_graph.py:384     _parse_attractions     ← 自由文本 → Pydantic 对象的"收口"处
⑥ trip_planner_graph.py:244     _plan_itinerary        ← 规划 Agent 出最终 JSON
```

### 逐个断点该看什么

| # | 位置 | 断住后重点观察 | 能回答的问题 |
| --- | --- | --- | --- |
| ① | `trip.py:50` `workflow.plan_trip(request)` | `request` 各字段 | 前端传了什么 |
| ② | `graph.py:614` `final_state = self.graph.invoke(initial_state)` | `initial_state` | State 初始长什么样 |
| ③ | `graph.py:173` `result = self.attraction_agent.invoke(...)` | `query`、`result` | Agent 收到的指令 & 原始返回 |
| ④ | `graph.py:79` `_extract_agent_output` | `result["messages"]` 整个列表 | **★ LLM 调了哪个工具、传了什么参数、工具回了什么** |
| ⑤ | `graph.py:384` `_parse_attractions` | `response`、`json_str`、`data` | JSON 抽取/解析是否成功 |
| ⑥ | `graph.py:244` `_plan_itinerary` | `query`（喂给规划 Agent 的汇总）、`output` | 最终行程 JSON 质量 |

> 🎓 **导师提示**：**最该重点蹲守的是断点 ④**（`_extract_agent_output` 里的 `result["messages"]`）。
> 这个 `messages` 列表里完整记录了一次 ReAct 循环：`HumanMessage`（你的 query）→ `AIMessage`（带 `tool_calls`，**这就是 LLM 的"决策"！**）→ `ToolMessage`（工具返回）→ `AIMessage`（最终回答）。
> **看懂这一串，你就真正"亲眼看到"了 Agent 的思考过程**，而不是停留在抽象理解。

### 2.2 不打断点也能看：开 `debug=True`

如果你不想逐行调，`create_agent` 自带详细日志开关。把 📄 `langgraph_agents.py:158` 的：

```python
agent_graph = create_agent(model=llm, tools=tools, system_prompt=system_prompt, debug=False)
```

改成 `debug=True`，控制台会打印出每一步的状态转移、工具调用与返回，效果类似"文字版断点"。**四个 `create_agent` 调用都有这个参数**（景点/天气/酒店/规划）。

### 2.3 进阶：用 LangSmith 做可视化追踪

项目已内置 LangSmith 支持（`config.py:73`）。在 `.env` 里：

```bash
LANGCHAIN_TRACING=true
LANGCHAIN_API_KEY=ls__你的key
LANGCHAIN_PROJECT=trip-planner
```

之后到 [smith.langchain.com](https://smith.langchain.com) 就能看到**每个 Agent 的调用树、token 消耗、工具入参出参的可视化时间线**——这是工业界调试多 Agent 系统的标准做法，比 print 强太多。

### 2.4 调试速查表（常见现象 → 去哪看）

| 现象 | 最可能的位置 | 排查方向 |
| --- | --- | --- |
| 启动就报"配置验证失败" | `config.py:106` `validate_config` | `LLM_API_KEY` / `AMAP_API_KEY` 是否非空 |
| 景点/酒店列表总是空 | `graph.py:384/461` `_parse_xxx` | 看 `response` 是不是合法 JSON（断点⑤） |
| 拿到的是"著名景点1/2"假数据 | `amap_mcp_tools.py:236` | 工具降级到了 mock（`uvx`/Key 问题） |
| 行程能出但字段错乱 | `graph.py:487` `_parse_trip_plan` | 看规划 Agent 的 JSON（断点⑥） |
| 直接走了兜底计划 | `graph.py:290` `_check_error` | 看 `state["error"]` 是哪个分支写的 |

## 本阶段小结

1. **MVP 最小配置**：真正卡住启动的只有 `LLM_API_KEY` 和 `AMAP_API_KEY` 两项；靠 mock 兜底，**一个有效的 LLM Key 就能跑通全链路**。
2. **别名映射**：`.env` 写 `LLM_*`，代码读 `OPENAI_*`，由 `config.py` 自动翻译——这是头号大坑。
3. **看思考过程**：6 个断点里，**`_extract_agent_output` 的 `messages` 列表是金矿**；嫌麻烦就开 `debug=True` 或上 LangSmith。

### 阶段三思考题

> 1. 为什么"一个有效 LLM Key + 一个假 AMAP Key"就能跑通？请用阶段二的"三层兜底"解释完整链条。
> 2. 如果断点④看到 `AIMessage` 里**根本没有 `tool_calls`**（LLM 直接编了答案），问题最可能出在哪？（提示：提示词 / 模型是否支持 function calling）
> 3. `python run.py` 和 `uvicorn ... --reload` 两种启动方式，对"工具缓存单例"（`_cached_tools`）的热重载行为有何不同影响？
