## 🧠 模块一：Memory（记忆 / 上下文管理）

### 1.1 先纠正一个常见误解

很多教程里的 "Memory" 指的是「跨会话的长期记忆」（比如存进向量库、下次还记得你）。
**本项目没有这种长期记忆**——它是一个**单次请求(one-shot)**的规划系统：一次 HTTP 请求进来，跑完一张图，返回结果，状态即销毁。

所以本项目的"记忆"，准确说是 **单次工作流内的「共享状态 / 短期上下文」**。它由两套机制承载：

| 机制 | 承载什么 | 作用域 |
| --- | --- | --- |
| **`TripPlannerState`（工作流状态）** | 景点/天气/酒店/最终计划/错误等结构化中间结果 | 整张图的一次执行 |
| **`messages` 通道（对话历史）** | 各节点产生的对话消息 | 整张图的一次执行 |
| **Agent 内部的 `messages`** | 单个 Agent 一次 ReAct 循环内的 user→tool→ai 消息 | 单次 `agent.invoke()` |

> 🎓 **导师提示**：Agent 的"记忆"不一定是花哨的向量数据库。**一个在节点间传递的、结构化的 State 对象，本身就是最朴素也最可靠的"工作记忆"**。先理解这种"短期工作记忆"，再去学长期记忆，路才走得稳。

### 1.2 核心数据结构：`TripPlannerState`

📄 `backend/app/workflows/trip_planner_state.py:23`

```python
class TripPlannerState(TypedDict):
    # 输入
    request: TripRequest
    user_input: str

    # 中间结果（各研究节点写入）
    attractions: List[Attraction]
    weather_info: List[WeatherInfo]
    hotels: List[Hotel]

    # 智能体通信（短期对话记忆）
    messages: Annotated[List[Dict], add_messages]

    # 最终输出
    trip_plan: Optional[TripPlan]
    error: Annotated[Optional[str], update_error]
    current_step: Annotated[str, update_step]
```

这里有三个**必须吃透**的知识点：

#### ① 为什么用 `TypedDict` 而不是普通 class？

这是 LangGraph 的约定：State 必须是一个"字典型"结构，图的每个节点接收完整 State、返回一个**"局部更新字典"**，LangGraph 负责把它合并回全局 State。例如 `_search_attractions` 只 `return {"attractions": [...]}`，其余字段保持不变。

#### ② `Annotated[..., reducer]` 是并行安全的关键 ★

注意 `messages`、`error`、`current_step` 三个字段都带了 `Annotated[类型, reducer函数]`。
**reducer（归约器）决定"当多个并行节点同时写同一个字段时，如何合并"**——这正是阶段一提到的「3 个研究 Agent 并行」能成立的底层保障。

- `messages: Annotated[..., add_messages]`：用 LangGraph 内置的 `add_messages`，把各分支产生的消息**追加合并**（而不是互相覆盖）。
- `error: Annotated[..., update_error]`（`trip_planner_state.py:18`）：自定义 reducer，`return prev or new`——**保留第一个报错的分支**，避免并行分支的错误互相覆盖。
- `current_step: Annotated[..., update_step]`（`trip_planner_state.py:13`）：`return new`——总是取最新值。

> 🎓 **导师提示（重点）**：如果没有 reducer，两个并行节点同时返回 `messages`，LangGraph 不知道该信谁，会直接报错或覆盖。**reducer 是 LangGraph 支持并行/扇入的精髓**。这道题如果面试被问到"LangGraph 怎么处理并行节点的状态冲突"，答案就是它。

#### ③ 状态是怎么"流动"的？

```
create_initial_state()  →  全部字段初始化（trip_planner_graph.py:611）
        ↓
search_attractions 返回 {"attractions":[...], "messages":[...]}  ─┐
check_weather      返回 {"weather_info":[...], "messages":[...]}  ─┤ 三者并行，reducer 合并
find_hotels        返回 {"hotels":[...], "messages":[...]}        ─┘
        ↓
join_research_results 返回 {"current_step":"research_completed"}
        ↓
plan_itinerary 读取 state["attractions"/"weather_info"/"hotels"] → 返回 {"trip_plan":...}
```

> 这也回答了阶段一思考题 1：**`plan_itinerary` 必须等三个研究节点都写完 State，因为它要读取它们写入的 `attractions/weather_info/hotels`**。数据依赖决定了执行顺序。

### 1.3 还有一组"没被用上"的辅助函数

`trip_planner_state.py:59-116` 定义了一批 `update_state_with_xxx` 函数（手动构造新 State）。但在新版图里，节点是**直接 `return 局部字典`** 交给 LangGraph 合并的，这些函数实际已被取代——属于迁移过程中的遗留代码，阅读时可忽略，但能帮你理解"作者从手动管理 State 演进到依赖 reducer"的思路。

## 🛠️ 模块二：Tools / Actions（工具定义与调用）

这是本项目最有"工业含量"的部分，核心文件 📄 `backend/app/tools/amap_mcp_tools.py`。

### 2.1 工具从哪来：MCP 协议动态加载

项目**没有手写**高德地图的 HTTP 调用，而是通过 **MCP（Model Context Protocol）** 把一个独立的 `amap-mcp-server` 进程的能力，自动转成 LangChain 工具。

📄 `amap_mcp_tools.py:100`

```python
connection = {
    "command": "uvx",                       # 用 uvx 启动一个子进程
    "args": ["amap-mcp-server"],            # 高德官方 MCP 服务器
    "transport": "stdio",                   # 通过标准输入输出通信
    "env": {"AMAP_MAPS_API_KEY": settings.amap_api_key}
}
tools = await load_mcp_tools(              # langchain_mcp_adapters 提供
    session=None, connection=connection,
    server_name="amap", tool_name_prefix=False
)
```

> 🎓 **导师提示**：MCP 的价值在于**解耦**。Agent 不关心"高德的 API 长什么样"，只看到一组标准工具（`maps_text_search`、`maps_weather`...）。想换成 Google Maps？换个 MCP server 即可，Agent 代码一行不改。这就是"可插拔工具生态"。

### 2.2 一个工程难点：异步工具 → 同步调用

MCP 工具是**异步**的（`_arun`），但 LangGraph 的工具节点常常在**同步**上下文里被调用。作者写了一个 `wrap_async_tools` 来桥接（`amap_mcp_tools.py:23`）：

```python
class SyncWrapper(tool.__class__):
    def _run(self, *args, **kwargs):
        nest_asyncio.apply()                 # 允许在已有事件循环里再跑
        return asyncio.run(self._arun(*args, **kwargs))
```

- `nest_asyncio` 解决"事件循环已在运行、不能再 `asyncio.run`"的经典报错；
- 还兼容了"已有运行中循环"的场景（`run_coroutine_threadsafe`）。

> 🎓 **导师提示**：这段代码你**初学时不必完全看懂**，但要记住一个工业现实——**"异步库 + 同步框架"的胶水代码，是真实 Agent 项目里最常见也最磨人的坑**。知道有这么个东西，将来踩坑时能想起来。

### 2.3 健壮性王牌：三层兜底加载

📄 `amap_mcp_tools.py:236` `get_cached_amap_tools()`

```
真实 MCP 全量工具  get_amap_mcp_tools()
   ↓ 失败
精简工具（只要 text_search + weather）  get_amap_essential_tools()
   ↓ 失败
模拟工具（返回写死的假数据）  create_mock_tools()
```

`create_mock_tools`（`amap_mcp_tools.py:279`）用 `@tool` 装饰器手写了 3 个假工具，返回固定 JSON。
**意义**：哪怕你没有高德 Key、网络不通，整条工作流依然能跑通、能调试。这是非常值得学习的"可降级"设计。

### 2.4 工具是如何被 LLM 触发的：Function Calling

工具与 Agent 的"接线"发生在 📄 `agents/langgraph_agents.py:154`：

```python
agent_graph = create_agent(
    model=llm,                # ChatOpenAI
    tools=tools,              # 上面加载的 MCP 工具
    system_prompt=system_prompt,
)
```

LangChain 的 `create_agent` 内部把工具的 schema 通过 **OpenAI 原生 Function Calling** 注入给 LLM。触发链路是：

```
节点调用 agent.invoke({"messages":[{"role":"user","content":query}]})
   ↓
LLM 看到工具 schema，自主决定输出一个 tool_call（含工具名 + 参数）
   ↓
create_agent 内置的 ToolNode 执行对应工具的 _run() → 拿到 JSON 结果
   ↓
结果作为 tool 消息回灌给 LLM
   ↓
LLM 产出最终回答（按提示词要求：直接吐出工具返回的原始 JSON）
```

最后节点用 `_extract_agent_output`（`trip_planner_graph.py:73`）从返回的 `messages` 里捞出最后一条 assistant 消息文本，再用 `_extract_json` + `_parse_xxx` 解析成 Pydantic 对象。

## ⚙️ 模块三：Planning / Reasoning（思考决策策略）

### 3.1 整体策略：Prompt 分工 + ReAct

本项目的"思考"不是靠一个大而全的提示词，而是 **"角色分工 + 各自 ReAct"**：

- **3 个研究 Agent**（景点/天气/酒店）采用 **ReAct 范式**：Reason（想该调哪个工具）→ Act（调工具）→ Observe（看结果）→ 输出。这个循环由 `create_agent` 自动驱动，**作者无需手写循环**。
- **1 个规划 Agent**（planner）**不带工具**，采用 **"强约束结构化输出（Structured Output via Prompt）"** 策略——它的任务不是探索，而是把已有信息"填模板"成严格 JSON。

### 3.2 研究 Agent 的提示词策略

📄 `langgraph_agents.py:14`（以景点 Agent 为例）

```
你是景点搜索专家。...
**重要提示:** 你必须使用工具来搜索景点!不要自己编造景点信息!
**可用工具:** {tools}      ← 运行时被 format 成工具列表
**工具调用要求:**
4. 工具调用完成后，请直接返回工具返回的原始JSON数据，不要进行总结或格式化
5. 返回的JSON应包含 name、address、location、visit_duration... 字段
```

三个设计意图，值得品味：
1. **强制用工具**（"不要编造"）——压制 LLM 幻觉，逼它走 Function Calling；
2. **`{tools}` 占位符**在 `create_agent` 前被 `system_prompt.format(tools=...)` 填充（`langgraph_agents.py:150`），让 LLM 知道有哪些工具；
3. **"直接返回原始 JSON、不要总结"**——这是关键技巧：让 Agent 输出**可被代码解析的结构**，而不是给人看的散文。把"LLM 的自由"约束在"机器可读"的边界内。

### 3.3 规划 Agent 的提示词策略：填模板

📄 `langgraph_agents.py:63` `PLANNER_AGENT_PROMPT`

它直接在提示词里塞了一整段**期望的 JSON 骨架**（`{{...}}`，双花括号是为了躲开 `.format` 的转义），并附 7 条硬性规则（温度必须纯数字、每天 2-3 景点、必须含三餐、必须含预算...）。

> 🎓 **导师提示**：这是 LLM 时代"**结构化输出**"最朴素也最常用的手法——**把目标 schema 直接示范给模型看**。虽然不如 `with_structured_output`/JSON mode 严谨，但胜在直观、可控、跨模型通用。配合后端 `_parse_trip_plan` 的容错解析 + 兜底计划，整体是"够用且健壮"的工程取舍。

### 3.4 ★ 新旧两版对照：看懂"工业级演进" ★

项目里保留了旧版实现 📄 `agents/old_helloagent_planner_agent.py`，对照阅读极有价值：

| 维度 | 旧版 (`old_helloagent_planner_agent.py`) | 新版 (`langgraph_agents.py` + LangGraph) |
| --- | --- | --- |
| **工具调用方式** | **手写文本协议** `[TOOL_CALL:amap_xxx:keywords=..,city=..]`，再用字符串解析触发 | **原生 Function Calling**，LLM 直接输出结构化 tool_call |
| **思考循环** | 自己串 4 个步骤（顺序执行 `agent.run()`） | `create_agent` 内置 ReAct 循环 + LangGraph 图编排 |
| **执行方式** | **串行**（步骤1→2→3→4） | **并行**（3 研究 Agent 扇出，再扇入规划） |
| **状态管理** | 普通变量传参 | `TripPlannerState` + reducer |
| **错误处理** | try/except + fallback | 专门的 `handle_error` 节点 + 条件边 |
| **可观测性** | print 日志 | 可接 LangSmith 追踪 |

> 🎓 **导师提示（重点）**：这张表就是一部浓缩的"Agent 工程进化史"。
> 旧版的 `[TOOL_CALL:...]` 文本协议是**早期没有 Function Calling 时的无奈之举**——靠提示词约定格式、靠正则解析，极其脆弱（LLM 少打个括号就崩）。
> 新版改用**模型原生工具调用 + 框架托管循环**，把"调用约定"交给模型和框架，开发者只管定义工具和提示词。**理解这个演进，比记住任何 API 都重要。**

## 本阶段小结

| 要素 | 本项目的实现 | 一句话本质 |
| --- | --- | --- |
| 🧠 Memory | `TripPlannerState`(TypedDict) + `add_messages`/自定义 reducer | "带归约规则的共享工作内存"，无长期记忆 |
| 🛠️ Tools | MCP 动态加载 + 异步转同步包装 + 三层兜底 + 原生 Function Calling | "可插拔、可降级的标准工具生态" |
| ⚙️ Planning | 角色分工提示词 + 研究 Agent ReAct + 规划 Agent 填 JSON 模板 | "把 LLM 的自由约束在机器可读的边界内" |

### 阶段二思考题

> 1. 如果我想让 Agent **记住用户上一次的旅行偏好**（跨请求长期记忆），现有架构缺了什么？应该在哪一层加？（提示：State 的生命周期 + 外部存储）
