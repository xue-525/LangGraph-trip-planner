## 1. 实战魔改：给 Agent 加一个「自定义工具」

我们以 **"查询本地文件 / 自定义天气"** 为例，走一遍**端到端的标准流程**。本项目工具的底层是 LangChain 的 `BaseTool`，最简单的造法是 `@tool` 装饰器（项目里 `create_mock_tools` 就是这么写的，`amap_mcp_tools.py:283`）。

### 1.1 在哪个目录写？—— `backend/app/tools/`

工具是 Agent 的"手脚"，统一放在 📁 `backend/app/tools/`。建议**新建一个文件**而不是塞进 `amap_mcp_tools.py`，保持单一职责。

```
backend/app/tools/
├── amap_mcp_tools.py        # 已有：高德 MCP 工具
└── custom_tools.py          # ← 新建：你的自定义工具
```

### 1.2 标准模板（直接可用）

📄 新建 `backend/app/tools/custom_tools.py`：

```python
"""自定义工具集"""

import json
import logging
from pathlib import Path
from typing import List
from langchain_core.tools import BaseTool, tool
from pydantic import BaseModel, Field

logger = logging.getLogger(__name__)


# ① 用 Pydantic 定义入参 schema —— 这就是 LLM 看到的"工具说明书"
class ReadLocalFileInput(BaseModel):
    """读取本地文件的入参"""
    filename: str = Field(description="要读取的文件名，如 'notes.txt'")
    max_chars: int = Field(default=2000, description="最多读取的字符数")


# ② 用 @tool 装饰器定义工具本体
#    - 第一个参数是工具名（LLM 调用时用的标识）
#    - docstring 至关重要：它是 LLM 判断"何时该用这个工具"的依据
#    - args_schema 绑定上面的入参模型
@tool("read_local_file", args_schema=ReadLocalFileInput)
def read_local_file(filename: str, max_chars: int = 2000) -> str:
    """读取项目 data 目录下的本地文本文件内容。当用户想查看本地资料、笔记或攻略时使用。"""
    logger.info(f"📂 读取本地文件: {filename}")
    try:
        base_dir = Path(__file__).resolve().parents[2] / "data"   # backend/data/
        target = (base_dir / filename).resolve()

        # 安全校验：禁止路径穿越，必须落在 data 目录内
        if not str(target).startswith(str(base_dir.resolve())):
            return json.dumps({"error": "非法路径"}, ensure_ascii=False)
        if not target.exists():
            return json.dumps({"error": f"文件不存在: {filename}"}, ensure_ascii=False)

        content = target.read_text(encoding="utf-8")[:max_chars]
        # ③ 关键：返回值用 str（建议 JSON 字符串），方便上层 _extract_json 解析
        return json.dumps({"filename": filename, "content": content}, ensure_ascii=False)
    except Exception as e:
        logger.error(f"读取文件失败: {e}", exc_info=True)
        return json.dumps({"error": str(e)}, ensure_ascii=False)


def get_custom_tools() -> List[BaseTool]:
    """对外暴露自定义工具列表（与 amap 工具一样可被注入 Agent）"""
    return [read_local_file]
```

### 1.3 工具规范要点（必须遵守）

| 规范 | 为什么 |
| --- | --- |
| **工具名** 用小写蛇形，语义清晰 | LLM 靠名字+描述选工具 |
| **docstring / Field description 写清"做什么、何时用"** | 这是 LLM 的唯一决策依据，写不好 = 工具不会被调用 |
| **入参用 Pydantic `BaseModel` + `args_schema`** | 框架据此生成 Function Calling schema |
| **返回值用 `str`（推荐 JSON 字符串）** | 上层 `_extract_json` / `_parse_xxx` 才能解析（见阶段二） |
| **内部自己兜底 try/except，错误也返回字符串** | 工具抛异常会中断整个节点，返回 `{"error":...}` 更安全 |

### 1.4 怎么"接线"让 Agent 用上它？—— 两种场景

#### 场景 A：让**已有 Agent**多一个工具（最简单）

工具最终是在 `TripPlannerWorkflow.__init__` 里通过 `get_cached_amap_tools()` 加载、再传给各 Agent 的（`trip_planner_graph.py:37-50`）。只要把你的工具**合并进工具列表**即可：

📄 改 `trip_planner_graph.py:37` 附近：

```python
from ..tools.amap_mcp_tools import get_cached_amap_tools
from ..tools.custom_tools import get_custom_tools          # ← 新增

# ...
self.tools = get_cached_amap_tools() + get_custom_tools()  # ← 合并
```

这样景点/天气/酒店 Agent 都会"看到"`read_local_file`，并在合适时机自主调用。**不需要改任何提示词或循环逻辑**——这正是 Function Calling 架构的威力。

#### 场景 B：为新工具配一个**专职 Agent + 新节点**（进阶）

如果这个工具值得一个独立步骤（比如"读取用户上传的旅行偏好文件"），就照葫芦画瓢加一条流水线，分 3 步：

**Step 1** — 在 `langgraph_agents.py` 加一个 Agent 工厂（仿 `create_attraction_search_agent`）：

```python
NOTE_AGENT_PROMPT = """你是资料读取专家，必须使用工具读取本地文件，不要编造内容。
**可用工具:** {tools}
读取后请直接返回工具的原始 JSON。"""

def create_note_agent(tools):
    llm = get_llm()
    desc = "\n".join(f"- {t.name}: {t.description}" for t in tools)
    return create_agent(model=llm, tools=tools,
                        system_prompt=NOTE_AGENT_PROMPT.format(tools=desc))
```

**Step 2** — 在 `trip_planner_graph.py` 加一个节点方法（仿 `_search_attractions`）：

```python
def _read_notes(self, state: TripPlannerState) -> Dict[str, Any]:
    try:
        result = self.note_agent.invoke(self._prepare_agent_input("读取 notes.txt", []))
        output = self._extract_agent_output(result)
        return {"messages": [{"role": "assistant", "content": output}]}
    except Exception as e:
        return {"error": f"读取资料失败: {e}", "current_step": "error"}
```

**Step 3** — 在 `_build_graph()`（`trip_planner_graph.py:126`）把节点挂上图，并接到并行扇出里：

```python
workflow.add_node("read_notes", self._read_notes)
workflow.add_edge(START, "read_notes")                              # 加入并行
workflow.add_edge(["search_attractions", "check_weather",
                   "find_hotels", "read_notes"],                    # ← 扇入加上它
                  "join_research_results")
```

> 🎓 **导师提示**：注意场景 B 要**同步更新三处**——Agent 工厂、节点函数、图的边（扇出 + 扇入）。如果你只加了节点没改扇入的 `add_edge` 列表，`join_research_results` 不会等它，新节点的结果可能丢失。**"加节点必同步改边"是 LangGraph 开发的肌肉记忆。** 别忘了若新节点要写 State 新字段，还需在 `TripPlannerState` 里声明（必要时配 reducer）。

## 2. 落地建议：从零写一个轻量级 Agent，先写哪几个类？

如果你想脱离这个项目、自己从零搭一个**最小可用**的 Agent，不必一上来就引入 LangGraph。建议按下面的"最小内核四件套"动手，由内到外：

### 2.1 最小内核：4 个类（按编写顺序）

```
① LLMClient   ──被──>  ② Tool  ──被──>  ③ Agent  ──编排──>  ④ Orchestrator
   (大脑)               (手脚)            (单体智能)          (多体协作, 可选)
```

| 顺序 | 类 | 职责 | 本项目对应 |
| --- | --- | --- | --- |
| ① | **`LLMClient`** | 封装 LLM 调用，统一 `chat(messages) -> response`，含 base_url/key/重试 | `llm_service.get_llm()`（用了 `ChatOpenAI`） |
| ② | **`Tool`** | 工具抽象：`name` + `description` + `args_schema` + `run(args)` | `BaseTool` / `@tool` |
| ③ | **`Agent`** | 核心循环：组装提示词 → 调 LLM → 若返回 tool_call 则执行工具 → 把结果回灌 → 直到出最终答案 | `create_agent` 帮你内置了这个循环 |
| ④ | **`Orchestrator`** | （多 Agent 才需要）定义多个 Agent 的协作顺序、共享状态、并行/汇合 | `TripPlannerWorkflow` + `StateGraph` |

### 2.2 Agent 核心循环伪代码（③ 最该先吃透）

```python
class Agent:
    def __init__(self, llm, tools, system_prompt):
        self.llm = llm
        self.tools = {t.name: t for t in tools}
        self.messages = [{"role": "system", "content": system_prompt}]

    def run(self, user_input: str, max_steps: int = 5) -> str:
        self.messages.append({"role": "user", "content": user_input})
        for _ in range(max_steps):                       # ← 防死循环，必须有上限
            resp = self.llm.chat(self.messages, tools=self.tools.values())
            if resp.tool_calls:                          # LLM 决定调工具 (Reason→Act)
                for call in resp.tool_calls:
                    result = self.tools[call.name].run(call.args)   # 执行 (Act)
                    self.messages.append(tool_message(call.id, result))  # 回灌 (Observe)
                continue                                 # 带着观察结果再想一轮
            return resp.content                          # 没有 tool_call → 最终答案
        return "已达最大步数"
```

> 🎓 **导师提示**：这 ~15 行就是一切 ReAct Agent 的"心脏"——**循环 + 判断有没有 tool_call**。理解它，你就理解了 `create_agent` / LangGraph 帮你封装掉的东西。**先用裸代码手写一遍这个循环，再回头用框架，你会对框架的每个参数都心中有数。**

### 2.3 演进路线（什么时候才上 LangGraph）

```
阶段 A：单 Agent + 几个工具（上面 4 个类够了）          ← 先到这
   ↓ 当流程出现"分支/并行/需要持久状态"
阶段 B：引入状态对象 State，手动按步骤编排多个 Agent      ← old_helloagent 那种串行写法
   ↓ 当并行/扇入/条件路由/错误分支变复杂、手写易错
阶段 C：上 LangGraph，用 StateGraph + reducer 托管编排   ← 本项目现在的形态
```

> 🎓 **导师提示**：**不要为了用框架而用框架。** 单 Agent 跑通后再考虑编排，需要并行/条件分支了再上 LangGraph。本项目从 `old_helloagent_planner_agent.py`（串行手写）演进到 `trip_planner_graph.py`（LangGraph），正是这条路线的真实样本——你完全可以重走一遍。
